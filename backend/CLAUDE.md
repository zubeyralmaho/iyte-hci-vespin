## Project identity reminder

The project is **Vespin**. Image names are
**`vespin-api`** and **`vespin-migrate`**. Go module path uses `vespin`.

## Stack — locked

- Go **1.23**
- HTTP: **chi** (`github.com/go-chi/chi/v5`)
- DB driver: **pgx v5** (`github.com/jackc/pgx/v5`)
- Query layer: **sqlc** (`sqlc.dev`) — code generated into `internal/db`
- Migrations: **golang-migrate**, raw SQL in `internal/db/migrations/`
- Config: **caarlos0/env** (`github.com/caarlos0/env/v11`)
- Logging: stdlib **`log/slog`**
- JWT: **golang-jwt** (`github.com/golang-jwt/jwt/v5`)
- Password hashing: **bcrypt** (`golang.org/x/crypto/bcrypt`)
- Validation: **go-playground/validator/v10**

Do NOT add or substitute:
- Gin, Echo, Fiber, or any other HTTP framework
- GORM, Ent, Bun, sqlx (raw)
- Viper, koanf, godotenv
- zap, zerolog, logrus
- Casbin or other policy engines

**Rules about layout:**

- The ONLY `package main` is in `cmd/api/main.go`.
- Everything else is in `internal/` so it cannot be imported by external code.
- Domains never import each other. If two domains need shared logic, extract
  it to `internal/<shared-thing>/` or use the database via `internal/db`.
- The dependency graph flows: `cmd → server → domains → db, auth, httpx, config`.
  The leaf packages (`db`, `auth`, `httpx`, `config`) do NOT import domains.

## Wiring pattern

`cmd/api/main.go` constructs all dependencies and passes them to
`server.NewRouter(...)` via a `Deps` struct:

```go
// Schematic
func main() {
    cfg := config.Load()
    pool := db.OpenPool(cfg.DatabaseURL)
    queries := db.New(pool)
    tokens := auth.NewTokens(cfg.JWTSecret)
    authMW := auth.Middleware(tokens)

    r := server.NewRouter(server.Deps{
        AuthMW:          authMW,
        UserHandler:     users.NewHandler(queries, pool, tokens),
        DeviceHandler:   devices.NewHandler(queries),
        EQHandler:       eqprofiles.NewHandler(queries, pool),
        PartyHandler:    partysessions.NewHandler(queries, pool),
        FirmwareHandler: firmware.NewHandler(cfg),
    })
    http.ListenAndServe(cfg.Addr, r)
}
```

NO dependency injection container. NO reflection-based wiring. NO globals.

## Handler pattern

Each domain exposes a `*Handler` struct with a constructor and a
`RegisterRoutes(r chi.Router)` method:

```go
package devices

type Handler struct {
    queries *db.Queries
}

func NewHandler(q *db.Queries) *Handler {
    return &Handler{queries: q}
}

func (h *Handler) RegisterRoutes(r chi.Router) {
    r.Get("/", h.list)
    r.Post("/", h.create)
    r.Get("/{id}", h.get)
    r.Patch("/{id}", h.update)
    r.Delete("/{id}", h.delete)
}

func (h *Handler) list(w http.ResponseWriter, r *http.Request) {
    userID := auth.UserIDFromContext(r.Context())
    rows, err := h.queries.ListDevicesByUser(r.Context(), userID)
    if err != nil {
        httpx.WriteError(w, err)
        return
    }
    httpx.WriteJSON(w, http.StatusOK, ToListResponse(rows))
}
```

Rules:
- Handlers are methods on `*Handler`. Never global functions.
- `userID` and `role` come from `r.Context()` via the `auth` package, NOT from
  request bodies or path params.
- Always pass `r.Context()` to queries — never `context.Background()`.
- Always check the error from a query. Always return on error.
- Always go through `httpx.WriteJSON` / `httpx.WriteError`. Never call
  `w.Write` directly. Never call `json.NewEncoder(w).Encode` directly.

## Service layer policy

Thin services exist ONLY where there is real logic. Pure CRUD goes
handler → queries directly.

Services live in `service.go` within the domain package. They are
**package-level functions**, not methods on a struct:

```go
package users

func ConvertGuest(
    ctx context.Context,
    q *db.Queries,
    guestID uuid.UUID,
    email, password string,
) (db.User, error) {
    // ...
}
```

A service exists when:
- The operation requires a transaction
- The operation composes multiple queries
- The operation has a real business rule (e.g., copy-on-write, status transitions)

A service does NOT exist for:
- Single-query reads
- Single-query writes with no derived logic

Do NOT create empty service files "for symmetry." Add a service when it
earns its keep, never before.

## DTO naming

Inside a domain package, drop the domain prefix to avoid stuttering:

```go
// In package devices:
type CreateRequest struct { ... }       // Used as devices.CreateRequest
type UpdateRequest struct { ... }
type Response struct { ... }
type ListResponse struct { ... }

// In package eqprofiles:
type ForkRequest struct { ... }         // eqprofiles.ForkRequest
```

NOT `DeviceCreateRequest`, NOT `DeviceResponse` — the package name is part
of the type's full name.

DTOs live in `dto.go` within the domain package, alongside the conversion
helpers:

```go
func ToResponse(d db.Device) Response { ... }
func ToListResponse(rows []db.Device) []Response { ... }
```

NEVER return sqlc-generated rows directly. Always convert through DTOs.

## Validation

Request bodies are validated via `go-playground/validator` struct tags.
Decoding + validation happens in `httpx.DecodeJSON`:

```go
type CreateRequest struct {
    Name       string `json:"name"       validate:"required,min=1,max=100"`
    DeviceType string `json:"deviceType" validate:"required,oneof=vespin_classic vespin_mini vespin_pro"`
}

func (h *Handler) create(w http.ResponseWriter, r *http.Request) {
    var req CreateRequest
    if err := httpx.DecodeJSON(r, &req); err != nil {
        httpx.WriteError(w, err)
        return
    }
    // ... validation already happened, req is valid
}
```

The OpenAPI spec carries the **contract** (required fields, formats, enums).
The validator carries the **runtime enforcement**. The spec is intentionally
loose on `maxLength`-style constraints; the validator is strict. Do not try
to mirror every validator tag in the OpenAPI spec.

## Errors

### Sentinel errors per domain

Each domain defines its sentinel errors at the top of the package:

```go
package devices

var (
    ErrNotFound              = errors.New("device not found")
    ErrInvalidEQProfileRef   = errors.New("invalid EQ profile reference")
)
```

Wrap errors with `%w` when adding context:

```go
return fmt.Errorf("create device: %w", err)
```

### Error → HTTP mapping

The mapping from Go errors to HTTP status + error-envelope codes lives in
`internal/httpx/error.go`. Handlers call `httpx.WriteError(w, err)` and the
function decides what status and code to emit. Add a case to that function
when you add a new sentinel.

The error envelope shape is fixed:

```json
{
  "error": {
    "code": "snake_case_code",
    "message": "human-readable message",
    "details": { /* optional */ }
  }
}
```

NEVER write a different error shape. NEVER return plain-text errors. NEVER
return Go's default error string directly to the client.

### The error code vocabulary

Defined in `backend/api/openapi.yaml` as part of the contract. Current codes:

Generic: `validation_failed`, `unauthorized`, `forbidden`, `not_found`,
`conflict`, `internal_error`

Specific: `email_taken`, `already_registered`, `invalid_credentials`,
`system_preset_immutable`, `guest_endpoint_forbidden`,
`invalid_eq_profile_reference`, `not_a_system_preset`,
`invalid_device_reference`, `invalid_status_transition`,
`device_already_in_session`

Do NOT invent new codes ad-hoc. If you need a new code, add it to the spec
and to the httpx error mapping in one PR.

## Logging

Use stdlib `log/slog`:

```go
slog.InfoContext(ctx, "device created", "device_id", deviceID, "user_id", userID)
slog.ErrorContext(ctx, "create device failed", "error", err, "user_id", userID)
```

Rules:
- Always use the `*Context` variants when you have a `context.Context`.
- Structured key-value pairs, not formatted strings.
- Log level guidance:
  - `Debug`: dev-only diagnostic detail
  - `Info`: lifecycle events (server started, device created, session ended)
  - `Warn`: degraded behavior, retries, rate limits hit
  - `Error`: errors that propagate to the client as 5xx
  - NOT `Error` for 4xx — those are expected and noisy at error level
- Never log secrets: passwords, tokens, hashes, JWT contents, full request bodies.
- Never log a full SQL query.

## Database

### sqlc workflow

Queries live in `internal/db/queries/<domain>.sql`. Example:

```sql
-- name: GetDeviceByID :one
SELECT * FROM devices WHERE id = $1;

-- name: ListDevicesByUser :many
SELECT * FROM devices WHERE user_id = $1 ORDER BY created_at DESC;

-- name: CreateDevice :one
INSERT INTO devices (user_id, name, device_type, firmware_version, battery_level, is_connected)
VALUES ($1, $2, $3, $4, $5, $6)
RETURNING *;
```

After editing queries: `sqlc generate` (from `backend/internal/db/`).

Rules:
- All queries are in `.sql` files. No query strings inline in Go code.
- Use the annotation comment exactly: `-- name: Name :one|:many|:exec|:execrows`.
- Use positional parameters (`$1`, `$2`), not named.
- Use `RETURNING *` on INSERTs that need the created row.
- For updates that return the updated row, use `UPDATE ... RETURNING *`.

### Transactions

When an operation needs multiple statements, use a transaction:

```go
func ConvertGuest(ctx context.Context, pool *pgxpool.Pool, q *db.Queries, ...) error {
    tx, err := pool.Begin(ctx)
    if err != nil {
        return fmt.Errorf("begin tx: %w", err)
    }
    defer tx.Rollback(ctx) // no-op if commit succeeds

    qtx := q.WithTx(tx)
    // ... call qtx.X(ctx, ...), qtx.Y(ctx, ...), ...

    return tx.Commit(ctx)
}
```

Services that need transactions take a `*pgxpool.Pool` as a parameter in
addition to `*db.Queries`. The handler injects both.

### Migrations

- One change per migration file.
- File naming: `000001_description.up.sql` + `000001_description.down.sql`.
  Use leading zeros for sortability.
- ALWAYS write a `.down.sql` even if it's just a DROP. golang-migrate refuses
  to apply migrations without down files in strict mode.
- **NEVER edit a committed migration.** If a previous migration was wrong,
  add a new one with a higher number that corrects the state. This rule has
  no exceptions, ever.
- System EQ presets are seeded by a numbered migration, not by application code.

### Connection pool

The pool is configured in `internal/db/pool.go` with sane defaults:

```go
// Schematic
config.MaxConns = 10
config.MinConns = 2
config.MaxConnLifetime = 30 * time.Minute
config.MaxConnIdleTime = 5 * time.Minute
```

Adjust only if there's a measured reason. For HCI scope, defaults are fine.

## Auth

### JWT

- Library: `golang-jwt/jwt/v5`.
- Algorithm: **HS256** (symmetric, single secret). Do NOT switch to RS256
  for this project — it's overkill for HCI scope.
- Claims: `sub` (user ID), `role` (`guest` or `registered`), `iat`, `exp`.
- Expiry: **30 days**. No refresh tokens — when the JWT expires, the user
  logs in again.

### Auth middleware

The middleware:
1. Reads `Authorization: Bearer <token>` header.
2. Verifies signature and expiry.
3. Parses `sub` (user ID UUID) and `role`.
4. Puts them in `context.Context` via typed keys.
5. Calls `next.ServeHTTP`.

Public routes (`/auth/*`) bypass the middleware. Registered-only routes
(`/users/me/preferences`, `/firmware/check`) check the role in the handler
or via a per-route middleware that rejects guests.

### Reading auth in handlers

```go
userID := auth.UserIDFromContext(r.Context())   // uuid.UUID
role := auth.RoleFromContext(r.Context())       // string: "guest" or "registered"
```

Always go through these accessors. Never read the context key directly.

### Password hashing

`bcrypt` with default cost. Never store plaintext passwords. Never log password hashes.

## Configuration

`internal/config/config.go` defines a `Config` struct loaded via
`caarlos0/env`. Example:

```go
type Config struct {
    AppEnv       string `env:"APP_ENV"      envDefault:"development"`
    Addr         string `env:"APP_ADDR"     envDefault:":8080"`
    DatabaseURL  string `env:"DATABASE_URL" envRequired:"true"`
    JWTSecret    string `env:"JWT_SECRET"   envRequired:"true"`
    LogLevel     string `env:"LOG_LEVEL"    envDefault:"info"`
}
```

Rules:
- All config from env vars. No flags. No config files.
- Required values use `envRequired:"true"`. The app fails to start if missing.
- Secrets (JWT_SECRET, DATABASE_URL with password) come from env only.
  Never hardcode. Never commit. Never log the Config struct.

## Health check

`GET /healthz` returns 200 with body `{"status":"ok"}`. No auth. No DB query.
Caddy hits this endpoint after deploys to verify the API is up.

Do NOT make `/healthz` query the database. The point is "is the process
running and serving HTTP." A more sophisticated readiness probe could check
DB connectivity at `/readyz`, but that's not currently in scope.

## Testing

- Light. See root CLAUDE.md for the testing posture.
- Table-driven tests, Go idiom:

```go
func TestForkSystemPreset(t *testing.T) {
    tests := []struct {
        name      string
        input     ForkRequest
        wantError error
    }{
        {"happy path", ForkRequest{Name: "My Jazz", Bands: validBands}, nil},
        {"empty name", ForkRequest{Name: "", Bands: validBands}, ErrValidation},
        // ...
    }
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            // ...
        })
    }
}
```

- Do NOT generate tests for trivial CRUD handlers.
- Do NOT mock sqlc-generated code. If you need a database in a test, spin up
  a real one via `testcontainers-go`.
- Do NOT write tests that hit the network.

## Common patterns to copy

When adding a new domain (hypothetically — the scope is closed):

1. Create `internal/<domain>/` with `handler.go`, `dto.go`, and optionally `service.go`.
2. Add queries in `internal/db/queries/<domain>.sql`.
3. Run `sqlc generate`.
4. Add the domain's routes to `internal/server/router.go`.
5. Add request/response schemas to `backend/api/openapi.yaml`.
6. Add sentinel errors to the domain package, map them in `internal/httpx/error.go`.

## What never to do

- Never edit `internal/db/models.go` or `internal/db/*.sql.go` — these are
  sqlc-generated. Edit the `.sql` files and regenerate.
- Never modify a committed migration. Add a new one.
- Never return sqlc-generated rows from handlers. Convert through DTOs.
- Never write a non-`httpx` response. All responses go through `httpx.WriteJSON`
  or `httpx.WriteError`.
- Never log secrets, full request bodies, or full SQL.
- Never call `context.Background()` from a handler.
- Never use `panic` in handler code. Return errors.
