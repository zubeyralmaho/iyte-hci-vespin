# Architecture

## System Overview

Vespin follows a client-server architecture with a clear separation between the
mobile frontend and the backend API.

```text
┌─────────────────────┐         ┌──────────────────────┐
│   Expo / React      │  HTTP   │    Go API Server     │
│   Native App        │ ──────► │    (chi router)      │
│   (iOS / Android)   │  JSON   │                      │
└─────────────────────┘         └──────────┬───────────┘
                                           │
                                           │ pgx
                                           ▼
                                ┌──────────────────────┐
                                │   PostgreSQL 16      │
                                │                      │
                                └──────────────────────┘
```

In production, a Caddy reverse proxy sits in front of the API to handle TLS and
HTTP/3.

## Backend Architecture

The backend uses a layered architecture within each domain:

```text
HTTP Handler → Service Logic → Database (sqlc queries)
```

- **Handlers** parse HTTP requests, validate input, and write responses.
- **Service logic** enforces business rules and orchestrates database calls.
- **Database layer** uses sqlc-generated type-safe Go code from raw SQL queries.

### Key Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Router | chi | Lightweight, stdlib-compatible, middleware-friendly |
| ORM | None (sqlc) | Type-safe SQL without ORM overhead |
| DB driver | pgx | Native PostgreSQL driver with superior performance |
| Config | caarlos0/env | Simple struct-tag-based environment parsing |
| Logging | stdlib slog | Structured logging without external dependencies |

## Frontend Architecture

The frontend is an Expo managed-workflow React Native app with:

- **File-based routing** via Expo Router
- **Server state** managed by TanStack Query (caching, refetching)
- **Client state** managed by Zustand (auth tokens, local UI state)
- **API layer** auto-generated by Orval from the OpenAPI spec
- **Styling** via NativeWind (Tailwind CSS for React Native)

### Component Hierarchy

```text
src/components/ui/       → Low-level primitives (Button, AppText, Icon)
src/components/layout/   → Layout primitives (Screen)
src/features/            → Feature-specific screens and components
```

UI and layout components must not import from `src/features/`.

## Domain Boundaries

| Domain | Responsibility |
|--------|---------------|
| Auth | Guest creation, registration, login, guest-to-registered conversion |
| Users | Profile retrieval, preferences (registered only) |
| Devices | CRUD for paired speakers, simulated connectivity |
| EQ Profiles | System presets, custom profiles, fork (copy-on-write) |
| Party Sessions | Multi-device session state and device membership |
| Firmware | Simulated version check (registered only) |

## Simulation Boundary

All hardware-facing aspects are simulated. The following fields are seeded at
creation time and never updated from real devices:

- `devices.firmware_version` — static string
- `devices.battery_level` — pseudo-random integer 20–100
- `devices.is_connected` — pseudo-random boolean

There is no real Bluetooth, audio streaming, firmware OTA, WebSocket, or push
notification support. These are explicitly out of scope.
